# `patterns-catalog.yaml` schema (v2)

Top-level machine-readable index of every cognitive unit in this repo, in three cohorts: **patterns** (flow shapes), **primitives** (building blocks the agent uses), and **modifiers** (transformations layered on a pattern). Aggregated from per-entry `metadata.json` plus tier-file presence on disk plus the composition matrix. Auto-generated by [`meta/validate-metadata.js --emit`](./meta/validate-metadata.js); never edit by hand.

The primary consumer is **agent-deployments' CI**, which fetches this file at generation time and embeds the relevant fields into its own `catalog.yaml`. Downstream tools (e.g. `agent-scaffold`) read the deployments catalog, not this one — so the contract surface that needs to stay stable is narrow.

> **What changed in v2.** The previous schema mashed flow shapes, building blocks, and transforms together under `patterns[]`. v2 splits them into three top-level keys (`patterns[]`, `primitives[]`, `modifiers[]`) so consumers can present them as orthogonal picker tiers. Workflows fold into `patterns[]` with `category: workflow`. The legacy `workflows[]` key is preserved as a derived view (`patterns[]` filtered by `category=workflow`) for one release cycle to give downstream consumers time to migrate.

## Regenerating

```bash
node meta/validate-metadata.js --emit patterns-catalog.yaml
```

The generator is deterministic: same source files → byte-identical output. The drift workflow (`.github/workflows/catalog-drift.yml`) regenerates on every PR and fails if the committed file diverges.

## Top-level shape

```yaml
schema_version: 2
generator_version: 2.0.0

patterns:       # flow shapes — 12 entries (8 agent + 4 workflow, by category)
  - id: react
    name: ReAct
    category: agent           # 'agent' (LLM-controlled) or 'workflow' (code-controlled)
    complexity: Intermediate
    description: "..."
    dir: patterns/react
    tier_files:
      overview: patterns/react/overview.md
      design: patterns/react/design.md
      implementation: patterns/react/implementation.md
      evolution: patterns/react/evolution.md
      observability: patterns/react/observability.md
      cost-and-latency: patterns/react/cost-and-latency.md
    evolvesFrom: [prompt-chaining]
    composableWith: [memory, reflection, rag]
    requires: [tools]
    tags: [reasoning, tool_use, loop, adaptive, open-ended]
    costTier: medium
    latencyTier: variable
    extras:
      prompts: patterns/react/prompts/
      schemas: patterns/react/schemas/
      code: patterns/react/code/

workflows:      # DERIVED VIEW — patterns[] filtered to category=workflow
                # Kept for one release cycle as backward compat; will be
                # removed in v3.
  - id: prompt-chaining
    name: Prompt Chaining
    category: workflow
    # … same entry shape as the patterns[] entry it derives from.

primitives:     # building blocks the agent uses — 3 entries
  - id: tool_use
    name: Tool Use
    category: primitive
    complexity: Beginner
    description: "..."
    dir: primitives/tool_use
    tier_files: { ... }
    evolvesFrom: [prompt-chaining]
    composableWith: [react, rag, routing]
    tags: [tools, function-calling]
    costTier: low
    latencyTier: low

modifiers:      # transformations layered on a pattern — 1 entry
  - id: human_in_the_loop
    name: Human in the Loop
    category: modifier
    complexity: Intermediate
    description: "..."
    dir: modifiers/human_in_the_loop
    tier_files: { ... }
    appliesTo: [any]          # which patterns this modifier wraps (or 'any')
    composableWith: [saga, event_driven, tool_use]
    requires: [pause-resume, approver-surface, audit]
    costTier: low
    latencyTier: variable

compositions:   # cross-cohort relationships parsed from composition/combination-matrix.md
  - {a: rag, b: react, kind: natural, rationale: "ReAct uses retrieval as one of its tools"}
  - {a: react, b: memory, kind: natural, rationale: "Add session persistence"}
```

## Field semantics

### Top-level

| Field | Type | Notes |
|---|---|---|
| `schema_version` | int | Bumped on breaking field/semantic changes. Consumers refuse versions above their declared max. v2 introduces `primitives[]` and `modifiers[]`. |
| `generator_version` | string | Bumped manually when the generator's output format changes in a non-breaking way. Diagnostic only. |
| `patterns` | array | All flow-shape entries (agent + workflow). Sorted by `id`. Distinguished by the per-entry `category` field. |
| `workflows` | array | **Derived view** — `patterns[]` entries with `category=workflow`. Preserved for one release cycle as backward compat; consumers should walk `patterns[]` and filter by `category` instead. |
| `primitives` | array | Building blocks the agent uses (tool_use, memory, skills). Sorted by `id`. |
| `modifiers` | array | Transformations layered on a pattern (human_in_the_loop). Sorted by `id`. Each entry may carry `appliesTo` listing the pattern ids it can wrap. |
| `compositions` | array | Cross-cohort relationship edges from the matrix. Sorted by `(a, b)`. Edges can reference any id from any cohort. |

The catalog deliberately **omits** `source_commit_sha` and `generated_at`. Those would create non-determinism (commit SHA tail-eating; timestamp churn) and break the drift CI's byte-diff. Consumers that need the upstream commit query it via the GitHub API at fetch time.

### Entry shape (patterns / primitives / modifiers)

All three cohorts share the same entry shape — only the `category` value and a few cohort-specific optional fields (`appliesTo` on modifiers) differ.

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | string | yes | Matches the directory name; the canonical identifier. |
| `name` | string | yes | Display name (used as a bridge to the composition matrix). |
| `category` | enum | yes | `"agent"` or `"workflow"` (for `patterns[]`), `"primitive"` (for `primitives[]`), or `"modifier"` (for `modifiers[]`). |
| `complexity` | enum | yes | `"Beginner"`, `"Intermediate"`, or `"Advanced"`. |
| `description` | string | yes | One-sentence summary. |
| `dir` | string | yes | Repo-root-relative path to the entry directory. |
| `tier_files` | map | yes | Map of tier name → relative path. Only includes tiers whose `.md` file exists on disk. |
| `evolvesFrom` | string[] | no | IDs this evolved from. Cross-cohort references allowed. |
| `evolvesInto` | string[] | no | IDs that evolved FROM this entry. Cross-cohort allowed. |
| `composableWith` | string[] | no | IDs that compose naturally with this one. Cross-cohort allowed. |
| `appliesTo` | string[] | no | **Modifiers only.** Pattern ids this modifier wraps, or `["any"]` for universal modifiers. |
| `requires` | string[] | no | Domain prerequisites (e.g. `"tools"`, `"vector-store"`). |
| `tags` | string[] | no | Searchable keywords. |
| `costTier` | string | no | `"low"`, `"low-medium"`, `"medium"`, `"high"`, `"variable"`. |
| `latencyTier` | string | no | Same vocabulary as `costTier`. |
| `extras` | map | no | Best-effort detection of optional subdirs (`prompts/`, `schemas/`, `code/`, `examples/`). Only present when at least one is detected. |

### Composition edge

| Field | Type | Required | Notes |
|---|---|---|---|
| `a` | string | yes | First pattern id, lexicographically smaller of the pair. |
| `b` | string | yes | Second pattern id, lexicographically larger of the pair. |
| `kind` | enum | yes | `"natural"`, `"useful"`, `"complex"`, `"redundant"`, or `"anti"`. |
| `rationale` | string | no | One-line explanation, lifted from the matrix cell. Omitted when the cell has only `**Kind**` with no em-dash explanation. |

Edges are undirected: each unordered pair appears at most once, with `a < b` lexicographically.

## Schema version policy

- **v1** (retired) — single `patterns[]` + `workflows[]` shape.
- **v2** (this release) — three-tier taxonomy. Adds `primitives[]` and `modifiers[]` top-level keys; the `workflows[]` key becomes a derived view of `patterns[]` for backward compat. New per-entry fields: `appliesTo` (on modifiers).
- Additive fields don't bump `schema_version`. Consumers must use forward-compatible parsing (ignore unknown fields).
- Removing a field, changing a field's type, or changing semantics bumps `schema_version`. Coordinate version bumps with the `agent-deployments` maintainer so deployments-CI updates in lockstep.

## Consumer contract

The primary consumer is `agent-deployments/scripts/generate_catalog.py`:

1. Fetches `https://raw.githubusercontent.com/jagguvarma15/agent-blueprints/main/patterns-catalog.yaml`.
2. Queries GitHub Commits API for the HEAD SHA on `main`; stamps as `blueprints.upstream_sha` in the deployments catalog.
3. Embeds `patterns[]`, `workflows[]`, and `compositions[]` into deployments' own `catalog.yaml`.

Downstream consumers (`agent-scaffold` etc.) read the deployments catalog, not this file directly.

## Drift CI

`.github/workflows/catalog-drift.yml` runs on every PR and push to `main`:

```yaml
- run: node meta/validate-metadata.js --emit /tmp/patterns-catalog.yaml
- run: |
    if ! diff -u patterns-catalog.yaml /tmp/patterns-catalog.yaml; then
      echo "::error::patterns-catalog.yaml is stale. Run 'node meta/validate-metadata.js --emit patterns-catalog.yaml' and commit."
      exit 1
    fi
```

The drift check is a hard gate: the only way to update the catalog is to regenerate it and commit the result.
