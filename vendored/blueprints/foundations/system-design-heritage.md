# System Design Heritage

Where the patterns in this repo inherit from classical distributed-systems work, and
which classical patterns are scoped to the sister `agent-deployments` repo instead of
here.

This is a positioning document, not a tutorial. Read it once to anchor the patterns
in their lineage; then carry the intuition you already have from systems work into the
LLM-pattern layer.

## Why this document exists

Blueprints in this repo didn't appear from nowhere. Many are LLM-shaped specializations
of structural patterns the systems-design community has refined for decades. Naming the
lineage does three things:

- Lets readers transfer intuition (failure modes, scaling shape, observability needs)
  from familiar systems work into agent design.
- Disambiguates the *cognitive* concerns this repo owns from the *operational* concerns
  the deployment-side repo owns — so reviewers don't expect Circuit Breakers here.
- Surfaces a small, deliberate gap list (four reliability patterns) that belong in
  `agent-deployments` rather than blueprints.

## Heritage map

Each blueprint in this repo specializes one or more classical patterns. The "what the
LLM layer adds" column is the reason the pattern earns its own page here rather than
being a footnote in a systems-design book.

### Workflows

| Blueprint | Classical roots | What the LLM layer adds |
|---|---|---|
| [Prompt Chaining](../patterns/prompt-chaining/overview.md) | Pipeline · Chain of Responsibility | LLM-mediated steps with validation gates between stages |
| [Parallel Calls](../patterns/parallel-calls/overview.md) | Scatter-Gather · MapReduce | Fan-out across independent prompts; aggregate at the end |
| [Orchestrator-Worker](../patterns/orchestrator-worker/overview.md) | Orchestrator + Worker Pool | The orchestrator is itself an LLM call that decomposes the task |
| [Evaluator-Optimizer](../patterns/evaluator-optimizer/overview.md) | Generate-and-Test · Feedback loop | The evaluator is an LLM; the optimizer is the same or a different LLM |

### Agent patterns

| Blueprint | Classical roots | What the LLM layer adds |
|---|---|---|
| [ReAct](../patterns/react/overview.md) | Sense-Plan-Act loop (robotics) | Sense → reason → tool call → observe, all driven by an LLM |
| [Plan & Execute](../patterns/plan_and_execute/overview.md) | Planner + Executor (workflow engines) | The plan is generated by an LLM upfront, not hand-authored |
| [Tool Use](../primitives/tool_use/overview.md) | Strategy · Adapter · RPC dispatch | Function schemas the LLM picks from at inference time |
| [Memory](../primitives/memory/overview.md) | Event Sourcing · CQRS (for audit/replay) | Short/long-term/semantic memory tiers fronted by an LLM reader |
| [RAG](../patterns/rag/overview.md) | Cache-Aside (with a semantic key) · Read-Through | Retrieval uses embeddings; the cache key is meaning, not a hash |
| [Reflection](../patterns/reflection/overview.md) | Generate-and-Test specialized to self-critique | The critic is another LLM call against measurable criteria |
| [Routing](../patterns/routing/overview.md) | API Gateway · Strategy · Front Controller | The router is an LLM classifier, not a static rule table |
| [Multi-Agent](../patterns/multi_agent/overview.md) | Orchestrator + Worker Pool · Pub-Sub | Workers are themselves agents; the supervisor is an LLM |
| [Event-Driven](../patterns/event_driven/overview.md) | Event-Driven Architecture · Pub-Sub · Message Queue | Consumers are agents; events trigger LLM-driven workflows |
| [Saga](../patterns/saga/overview.md) | Saga (same name, same shape) | Each step or its compensator may be an LLM call; the saga log carries reasoning traces |
| [Human in the Loop](../modifiers/human_in_the_loop/overview.md) | Approval workflow · 2PC-style commit gate | The proposal is generated by an agent; the approver is a human |

## What translates and what doesn't

The DesignGurus article *[50 System Design Patterns Every Engineer Should Know](https://designgurus.substack.com/p/50-system-design-patterns-every-engineer)*
catalogues storage, caching, communication, reliability, scaling, data processing,
API design, infrastructure, consistency, and observability patterns.

Roughly fifteen of the fifty map onto agent work:

- **Already first-class in this repo:** Saga, Event-Driven Architecture (and implicitly
  Pub-Sub and Message Queue, both subsumed by the Event-Driven pattern).
- **Inherited as structural ideas** (named in the heritage map above): Pipeline,
  Scatter-Gather, MapReduce, Event Sourcing, CQRS, Cache-Aside, API Gateway, Strategy.
- **Documented in `agent-deployments` as cross-cutting production concerns**:
  Circuit Breaker, Retry with Exponential Backoff, Bulkhead, Timeout, Idempotency,
  Dead Letter Queue, Distributed Tracing, Rate Limiting, Graceful Degradation.

The remaining thirty-something patterns (sharding, write-ahead logs, CDN, reverse
proxy, service mesh, BFF, vector clocks, quorum, consistent hashing, two-phase commit,
connection pooling, and so on) sit at infrastructure layers below where agent patterns
operate. They show up in the substrate the deployment-side specs assume — not as
blueprints in their own right.

## Reliability concerns: see `agent-deployments`

Four reliability concerns surface across nearly every pattern in this repo. They live
in the sister `agent-deployments` repo under `docs/cross-cutting/` because they're
operational concerns shared by every cognitive pattern — not new cognitive patterns
themselves. The deployment-side docs cover Python + TypeScript implementations,
combination order, anti-patterns, and tests.

1. **Circuit Breaker** — Stop calling a failing tool or sub-agent; return a fallback;
   probe periodically for recovery. Relevant to Tool Use, Multi-Agent, RAG (when the
   retrieval store is degraded), and Event-Driven (when a downstream consumer is
   wedged). See
   [`cross-cutting/resilience.md#circuit-breakers`](https://github.com/jagguvarma15/agent-deployments/blob/main/docs/cross-cutting/resilience.md#circuit-breakers).

2. **Retry with Exponential Backoff** — Retry transient LLM-call and tool-call
   failures with growing delays plus jitter; cap by attempt count and total budget.
   Foundational under almost every other pattern. See
   [`cross-cutting/resilience.md#retries`](https://github.com/jagguvarma15/agent-deployments/blob/main/docs/cross-cutting/resilience.md#retries)
   (the same file also covers Timeouts and Bulkheads and the order in which to
   compose them: bulkhead → timeout → retry → circuit breaker).

3. **Idempotency** — Design tool calls and agent actions so replays are safe. Saga
   compensation, Event-Driven consumers, and HITL approval replay all depend on
   this. See
   [`cross-cutting/idempotency.md`](https://github.com/jagguvarma15/agent-deployments/blob/main/docs/cross-cutting/idempotency.md)
   (two-phase claim/release, single-phase SETNX, unique-constraint dedup,
   outbound idempotency keys).

4. **Distributed Tracing** — One trace spanning the user request, every LLM call,
   every tool call, every sub-agent, and every async event. LLM-layer tracing lives
   in
   [`cross-cutting/observability.md`](https://github.com/jagguvarma15/agent-deployments/blob/main/docs/cross-cutting/observability.md)
   (Langfuse); cross-service tracing lives in
   [`stack/opentelemetry.md`](https://github.com/jagguvarma15/agent-deployments/blob/main/docs/stack/opentelemetry.md).
   The per-pattern `observability.md` files in this repo give pattern-specific
   guidance that sits on top.
