# Project layout

Canonical directory structure for projects generated by `agent-scaffold` from any recipe under [`docs/recipes/`](../recipes/). Recipes declare individual `required_files`; this document explains the surrounding tree they sit in and the conventions every file in it follows.

> Authoritative since: introduction of this file.
> Referenced from: every recipe under `docs/recipes/` (see "Key files" section in each recipe) and [`docs/recipes/SCHEMA.md`](../recipes/SCHEMA.md).
> Related: [`../capabilities/README.md`](../capabilities/README.md) (env-var contracts), [`testing-strategy.md`](testing-strategy.md), [`logging-structured.md`](logging-structured.md).

## Why this exists

Every recipe enumerates its `required_files` — `app/main.py`, `app/agent/researcher.py`, `tests/unit/...`. Until this document landed, the surrounding tree (where `pyproject.toml` lives, what `app/` looks like for a multi-agent project, where `.env.example` belongs, how `tests/` is sub-divided) was implicit. New recipes drifted; the `agent-scaffold` writer assumed conventions it never published.

This file makes those conventions explicit. Recipes link here for their "Project structure" / "Key files" sections; the scaffold writer enforces them when emitting a project.

## 1. Python tree

```
project-name/
├── app/
│   ├── __init__.py
│   ├── main.py                # FastAPI / uvicorn entry (or CLI entry)
│   ├── settings.py            # Pydantic-settings, reads .env
│   ├── agent/                 # Agent definitions
│   │   ├── __init__.py
│   │   └── <agent_name>.py
│   ├── api/                   # HTTP route handlers (if applicable)
│   │   └── <route>.py
│   ├── tools/                 # @agent.tool implementations
│   │   └── <tool_name>.py
│   ├── models/                # Pydantic request/response schemas
│   │   └── schemas.py
│   └── db/                    # SQLAlchemy models, session, migrations target
│       └── models.py
├── tests/
│   ├── unit/
│   ├── integration/
│   └── eval/
│       └── dataset.jsonl
├── pyproject.toml             # uv-managed; ruff + mypy + pytest config inline
├── .env.example               # every required env var, no secrets
├── .gitignore
├── Dockerfile
├── docker-compose.yml         # only if recipe declares external services
├── .github/
│   └── workflows/
│       └── ci.yml
└── README.md
```

- Single `app/` package (never `src/app/` or `<project_name>/app/`). The `app` namespace is reserved.
- `app/agent/` holds agent definitions; `app/api/` holds HTTP routing; `app/tools/` holds tools. Recipes with multi-agent topologies replace `agent/` with `agents/` (plural).
- Migrations and Alembic config live under `app/db/migrations/` when used.

## 2. TypeScript tree

```
project-name/
├── src/
│   ├── index.ts               # Hono / Node entry (or CLI entry)
│   ├── config.ts              # Zod-validated config from env
│   ├── agent/
│   │   └── <agentName>.ts
│   ├── api/
│   │   └── <route>.ts
│   ├── tools/
│   │   └── <tool-name>.ts
│   └── schemas.ts             # Zod schemas
├── tests/
│   ├── unit/
│   ├── integration/
│   └── eval/
│       └── dataset.jsonl
├── package.json               # pnpm-managed; "type": "module"
├── tsconfig.json
├── .env.example
├── .gitignore
├── Dockerfile
├── docker-compose.yml
├── .github/
│   └── workflows/
│       └── ci.yml
└── README.md
```

- ESM only (`"type": "module"` in `package.json`); no CommonJS.
- `src/` package; never `app/` for TypeScript.
- File names use kebab-case (`web-search.ts`); exported symbols use camelCase / PascalCase.

## 3. Files always required

Regardless of recipe, the scaffold writer enforces these:

| File | Purpose | Enforced where |
|------|---------|----------------|
| `README.md` | Project overview, run instructions, env vars | `agent-scaffold` validator's `required_files_on_disk` check |
| `.env.example` | Every required env var with `<replace-me>` placeholders | Same |
| `.gitignore` | At minimum `.env`, `__pycache__/`, `node_modules/`, build artifacts | Same |
| Entry point | `app/main.py` (Python) or `src/index.ts` (TypeScript) | Recipe `required_files` + smoke test |
| Manifest | `pyproject.toml` (Python) or `package.json` (TypeScript) | Validator's static-tier check |
| At least one smoke test | The recipe's stated smoke-check target | Validator's smoke-tier check |

Per-recipe additions land in the recipe's `required_files` list.

## 4. Env file rules

- `.env.example` lives at the project root, committed.
- `.env` is **never committed** (must appear in `.gitignore`).
- Every variable listed in `.env.example` carries a `<replace-me>` placeholder or a safe default — no real secrets.
- Variable names follow the canonical set declared in [`../capabilities/README.md`](../capabilities/README.md) (e.g., `ANTHROPIC_API_KEY`, `DATABASE_URL`, `REDIS_URL`, `LANGFUSE_PUBLIC_KEY`). When a capability owns a variable, the recipe inherits that name verbatim.
- Group vars by concern with `# ---- <concern> ----` comments so the file scans cleanly.

## 5. Test directory rules

Three subtrees, three contracts:

- **`tests/unit/`** — mirrors the source tree. Pure-function tests; no network, no DB, no LLM. `test_<module>.py` (Python) or `<module>.test.ts` (TypeScript). Must run in under 5 seconds for the suite.
- **`tests/integration/`** — exercises real-or-mocked external services. Requires the recipe's `external_services` to be up (or replaced with the canonical mock fixture). Typically uses `docker-compose up` services from the project root.
- **`tests/eval/`** — agent quality. Holds `dataset.jsonl` (input/expected pairs) plus the runner that walks the dataset. See [`testing-strategy.md`](testing-strategy.md) for evaluator selection and [`eval-data.md`](eval-data.md) for dataset generation, curation, and drift detection.

Recipes declare smoke-tier tests as `required_files` and own the dataset shape.

## 6. Monorepo layout

When one project owns multiple agents that share infrastructure (database, observability, deployment target), use this layout:

```
project-name/
├── agents/
│   ├── researcher/
│   │   ├── app/
│   │   ├── tests/
│   │   ├── Dockerfile
│   │   └── pyproject.toml      # workspace member
│   ├── reviewer/
│   │   └── …                   # mirrors researcher/
│   └── notifier/
│       └── …
├── infra/
│   ├── docker-compose.yml      # shared services (postgres, redis, langfuse)
│   ├── terraform/
│   └── migrations/
├── pyproject.toml              # uv workspace root
├── .env.example                # union of every agent's required vars
├── .github/workflows/ci.yml    # per-agent matrix
└── README.md
```

Apply this layout only when ≥2 agents share infra. Single-agent projects stay flat.

## 7. Docker layout

- **`Dockerfile`** lives at project root (single-agent) or per-agent under `agents/<name>/` (monorepo). Multi-stage build: `build` → `runtime`. Python uses `python:3.12-slim` + `uv`; TypeScript uses `node:20-alpine` + `pnpm`.
- **`docker-compose.yml`** lives at project root in both layouts. References capability service blocks from `docs/capabilities/<kind>/<service>.md` — the recipe imports those verbatim via `agent-scaffold`'s capability resolver. Do not edit service definitions inline.
- Services are named by capability `id` (`postgres`, `redis`, `qdrant`, `langfuse`) — never custom names — so cross-recipe tooling works.
- The compose file is only generated when the recipe declares external services or capabilities that require runtime infra.

| HTTP routes | `app/api/` | `src/api/` |
| Tools | `app/tools/` | `src/tools/` |
| Database models | `app/db/models.py` | `src/db/models.ts` |
| Migrations | `app/db/migrations/` (Alembic) | `src/db/migrations/` (Drizzle / Kysely) |
| Unit tests | `tests/unit/test_<mod>.py` | `tests/unit/<mod>.test.ts` |

## 9. Host port allocation

Single source of truth for the **host** ports each capability service binds. Two services must never bind the same host port in one `docker compose up`; capability docker fragments are pre-assigned here so any resolvable recipe stack is collision-free. The agent backend owns `8000`; everything that natively listens on `8000` is host-remapped off it.

| Service | Host port | In-container | URL env var |
|---------|-----------|--------------|-------------|
| Agent backend (app) | `8000` | `8000` | `APP_PORT` |
| Postgres | `5432` | `5432` | `DATABASE_URL` |
| Redis | `6379` | `6379` | `REDIS_URL` |
| Qdrant | `6333` (HTTP), `6334` (gRPC) | `6333` / `6334` | `QDRANT_URL` |
| Chroma | `8002` | `8000` | `CHROMA_URL` |
| Zep | `8003` | `8000` | `ZEP_API_URL` |
| Kafka | `9092` | `9092` | `KAFKA_BOOTSTRAP_SERVERS` |
| Langfuse | `3001` | `3000` | `LANGFUSE_HOST` |
| Grafana | `3002` | `3000` | — |
| Temporal | `7233` (gRPC), `8233` (UI) | `7233` / `8233` | `TEMPORAL_HOST_URL` |
| Containerized frontend (minimal-chat / Vite) | `3000` | `3000` | `VITE_AGENT_URL` → backend |
| Next.js chat (host `pnpm dev`) | `3000` | — | `NEXT_PUBLIC_AGENT_URL` → backend |
| Streamlit chat (host `streamlit run`) | `8501` | — | `AGENT_URL` → backend |

Rules:

- **Only the host port (left of `host:container`) is remapped** to avoid collisions. The container-internal port (right side) stays at the service's native port, so a service remapped on the host is unchanged inside the network.
- **In-network URLs use `service:<container-port>`** (e.g. `http://zep:8000`, `http://chroma:8000`), not the host port. An app running inside compose reaches a dependency by service name on its container port; only an app running on the **host** uses `localhost:<host-port>`.
- **Healthchecks run inside the container** and target the container port (e.g. Zep's `/healthz` and Chroma's `/api/v1/heartbeat` both probe `localhost:8000`), regardless of the host remap.
- The frontends never bind a host port simultaneously: `minimal-chat` is the only one served as a compose container (host `3000`, pinned by the scaffold's `normalize_frontend_service`); `nextjs-chat` and `streamlit` run as host dev processes and are not part of `docker compose up`.
